When should accessor methods be used?
To provide read-only access: Use them when you need to display an object's information (like printing a DVD's title in a Cart) without allowing external classes to modify the original data.

To maintain Encapsulation: They allow you to keep fields private. This protects the internal state while providing a controlled way to "export" data.

When there is a specific "Use Case": Only create a getter if it is strictly necessary for the program's logic (e.g., needing getCost() to calculate the totalCost() of a cart).

Core Principle: Follow the "Tell, Don't Ask" rule—minimize using getters to pull data out for external calculations; instead, tell the object to perform the action itself whenever possible.

Question: If you create a constructor to build a DVD by title and then another to build a DVD by category, does JAVA allow this? 


Answer: No. In Java, constructors are overloaded based on their signatures (the number, order, and type of parameters). Since both title and category are of type String, the two constructors would have the same signature DigitalVideoDisc(String), which causes a compilation error.
